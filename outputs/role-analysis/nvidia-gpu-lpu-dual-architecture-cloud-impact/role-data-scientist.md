## DS 요약 (5 bullets)

- **모델 호환성**: LPU는 Llama·Whisper 등 일부 transformer 계열만 지원 — MoE·비정형 아키텍처는 GPU 전용
- **파이프라인 전환**: vLLM/SGLang → Groq SDK 전환 시 정적 컴파일·동적 배치 제약으로 실험 반복 속도 저하
- **실험 설계**: TTFT·tok/s·배치 처리량을 종속 변수로 한 GPU vs LPU A/B 실험, 워크로드 유형별 분리 분석 필요
- **모니터링**: VictoriaMetrics에 LPU 전용 메트릭(TTFT p50/p95, 토큰 처리량, 대기 시간) 익스포터 추가
- **권고**: TTFT 민감 워크로드(에이전트·챗봇)는 LPU 우선, 배치·긴 시퀀스·미지원 모델은 GPU 유지

## 2. 모델 호환성: LPU 지원 범위와 제약

### 지원 모델 범위

| 아키텍처 유형 | LPU 지원 | GPU 지원 | DS 관점 제약 |
|---------------|----------|----------|--------------|
| **표준 Transformer** (Llama, Mistral) | 지원 | 지원 | LPU 우선 검토 가능 |
| **Whisper (Speech)** | 지원 | 지원 | 실시간 음성 추론에 LPU 유리 |
| **MoE (Mixtral 등)** | 제한적/미지원 | 지원 | GPU 전용, LPU 라우팅 제외 |
| **비정형/커스텀** | 미지원 | 지원 | GPU 전용 |
| **Quantized (INT8/INT4)** | Groq 정적 컴파일 | vLLM 동적 | LPU는 사전 컴파일 필요 |

### 핵심 제약

1. **정적 컴파일**: LPU는 모델별 사전 컴파일 필요 — 새 모델 실험 시 컴파일 대기 시간 발생
2. **동적 배치 제한**: 배치 크기·시퀀스 길이 유연성이 GPU 대비 낮음 — 실험 설계 시 고정 배치 전제
3. **모델 버전 관리**: LPU 지원 모델 목록과 GPU 모델 목록 이원화 — DS는 모델 선택 시 가속기 매핑 테이블 참조 필요

### 모델 선택 의사결정 매트릭스

```
모델 아키텍처 + 워크로드 특성 → 가속기 추천
├── TTFT 민감 (에이전트, 챗봇) + LPU 지원 모델 → LPU
├── 배치 추론 + 긴 시퀀스 → GPU
├── MoE/비정형 → GPU
└── 실시간 음성 (Whisper) → LPU
```

## 3. 추론 파이프라인 변화: vLLM/SGLang → Groq SDK 전환 포인트

### 현재 vs LPU 도입 후

| 단계 | GPU (vLLM/SGLang) | LPU (Groq SDK) | DS 영향 |
|------|-------------------|----------------|---------|
| **모델 로딩** | 동적, 런타임 선택 | 정적, 사전 컴파일 | 실험 반복 시 컴파일 오버헤드 |
| **배치 처리** | 동적 batching | 제한적 | 배치 크기 실험 설계 변경 |
| **추론 실행** | CUDA 커널 | Groq 전용 | API/SDK 호출 방식 상이 |
| **결과 수집** | 표준 메트릭 | Groq 전용 메트릭 | 통합 대시보드 설계 필요 |

### 전환 포인트 판단 기준

1. **TTFT SLO 미달 시**: GPU 50–100ms → LPU ~10ms, 에이전트형 워크로드에서 LPU 전환 검토
2. **단일 스트림 tok/s**: LPU 10배 우위 — 스트리밍 응답 품질 개선 시 LPU
3. **배치 처리량**: GPU 유리 — 대량 추론 작업은 GPU 유지
4. **모델 실험 주기**: LPU는 컴파일 대기로 실험 속도 저하 — 빠른 모델 A/B 테스트는 GPU 우선

### 파이프라인 분기 설계

```
추론 요청
    ├── 모델 ∈ LPU 지원 목록?
    │       ├── 예 → TTFT 민감? → LPU
    │       └── 아니오 → GPU
    └── 배치 크기 > N? → GPU (LPU 배치 제약)
```

## 4. 실험 설계: GPU vs LPU 성능 A/B 실험 방법론

### 실험 목표

- **H0**: GPU와 LPU 간 TTFT·tok/s·처리량에 유의한 차이 없음
- **H1**: LPU가 TTFT·단일 스트림 tok/s에서 유의하게 우수, GPU가 배치 처리량에서 우수

### 실험 설계

| 항목 | 설정 |
|------|------|
| **독립 변수** | 가속기 타입 (GPU vs LPU), 모델 (Llama-3-8B 등), 배치 크기 (1, 4, 16, 32) |
| **종속 변수** | TTFT (ms), tok/s, 배치당 처리량, p95 지연 시간 |
| **통제 변수** | 입력 시퀀스 길이, 출력 토큰 수, 동일 모델 버전 |
| **샘플 크기** | 각 조건당 최소 100회 반복 (통계적 검정력 0.8, α=0.05 가정) |
| **랜덤화** | 요청 순서 무작위, 네트워크/캐시 영향 최소화 |

### 통계 방법

1. **정규성 검정**: Shapiro-Wilk (TTFT, tok/s 분포)
2. **비교 검정**: 정규 분포 시 t-test, 비정규 시 Mann-Whitney U
3. **다중 비교**: Bonferroni 또는 FDR 보정 (가속기×모델×배치 조합)
4. **효과 크기**: Cohen's d 보고 — 실용적 의미 해석

### 실험 시나리오 매트릭스

| 시나리오 | 배치 | 시퀀스 | 예상 우위 |
|----------|------|--------|-----------|
| 에이전트 대화형 | 1 | 짧음 | LPU |
| 챗봇 스트리밍 | 1–4 | 중간 | LPU |
| 배치 요약 | 16–32 | 긴 | GPU |
| 실시간 음성 | 1 | 고정 | LPU |

## 5. 모니터링 지표: LPU 추론 품질 및 성능 (VictoriaMetrics 연동)

### 핵심 메트릭

| 메트릭 | 설명 | VictoriaMetrics 라벨 | SLO 예시 |
|--------|------|---------------------|-----------|
| `inference_ttft_seconds` | Time To First Token | `accelerator=lpu`, `model=` | p95 < 50ms |
| `inference_tokens_per_second` | 토큰 처리량 | `accelerator=lpu`, `model=` | p50 > 100 |
| `inference_batch_throughput` | 배치당 처리량 | `accelerator=lpu` | - |
| `inference_queue_wait_seconds` | 큐 대기 시간 | `accelerator=lpu` | p95 < 5s |
| `inference_error_rate` | OOM·타임아웃 비율 | `accelerator=lpu`, `error_type=` | < 0.1% |
| `inference_request_total` | 요청 수 | `accelerator=lpu`, `model=` | - |

### VictoriaMetrics 연동 설계

1. **LPU 익스포터**: Groq SDK 또는 추론 서비스에서 Prometheus 포맷 메트릭 노출
2. **VictoriaMetrics scrape**: LPU Pod/서비스 타겟 추가, `accelerator=lpu` 라벨 부여
3. **대시보드**: Grafana에서 GPU/LPU 분리 패널, TTFT·tok/s 시계열, 에러율 트렌드
4. **알림**: TTFT p95 SLO 초과, 에러율 스파이크 시 Alertmanager 규칙

### 데이터 품질 검증

- **결측치**: LPU 노드 장애 시 메트릭 결측 — 보간 또는 "unknown" 라벨
- **이상치**: 네트워크 지연·캐시 미스로 인한 TTFT 스파이크 — IQR 기반 이상치 플래그
- **드리프트**: 모델 버전 변경 시 메트릭 분포 변화 — 베이스라인 대비 drift 감지

## 6. DS 관점 권고: 모델 선택 및 추론 최적화 전략

### 즉시 분석 작업

1. **LPU 지원 모델 목록 정리**: Groq 공식 문서 기반 매핑 테이블 작성, DS 실험 시 참조
2. **벤치마크 스크립트**: TTFT·tok/s 재현 가능한 벤치마크 (고정 시드, 동일 입력)
3. **A/B 실험 인프라**: GPU vs LPU 라우팅을 Feature flag로 제어, 실험 그룹 할당 로직

### 모델 선택 전략

| 워크로드 | 권장 가속기 | 근거 |
|----------|-------------|------|
| 에이전트·챗봇 (TTFT 민감) | LPU | TTFT 5–10배 개선 |
| 배치 요약·번역 | GPU | 배치 처리량 우위 |
| 실시간 음성 (Whisper) | LPU | 단일 스트림 tok/s 우위 |
| MoE·커스텀 모델 | GPU | LPU 미지원 |
| 빠른 모델 실험·A/B | GPU | LPU 정적 컴파일 지연 |

### 추론 최적화 로드맵

1. **Phase 1**: GPU 기반 벤치마크 베이스라인 수립, SLO 기준 정의
2. **Phase 2**: LPU POC, 동일 모델·동일 입력으로 GPU vs LPU 비교 실험
3. **Phase 3**: 워크로드별 라우팅 정책 수립, VictoriaMetrics 대시보드 구축
4. **Phase 4**: 프로덕션 트래픽 분할 A/B 테스트, 통계적 유의성 검증 후 전환

### 데이터 인프라 개선

- **추론 로그 스키마**: `accelerator_type`, `model_id`, `ttft_ms`, `tokens_per_sec` 필드 표준화
- **실험 메타데이터**: 실험 ID·그룹·날짜를 로그에 포함 — 사후 분석·재현성 확보
- **NATS 연동**: 추론 이벤트를 NATS로 퍼블리시 — 실시간 모니터링·다운스트림 분석
