## 2. 신규 역량 요구: LPU 도입 시 채용 또는 육성이 필요한 역할/스킬

### 채용 또는 육성 대상 역할

| 역할 | 필요 역량 | 우선순위 | 확보 방안 |
|------|-----------|----------|-----------|
| **LPU 인프라 엔지니어** | Groq SDK, LPU 런타임, Device Plugin/CDI | P0 | 채용(희소) 또는 GPU 엔지니어 육성 |
| **하이브리드 스케줄링 전문가** | Slinky GRES, Kueue extended resource, 이종 가속기 스케줄링 | P0 | 내부 육성(기존 Slinky 운영자) |
| **추론 스택 통합 엔지니어** | vLLM-Groq, ONNX Runtime LPU 백엔드, GPU/LPU 라우팅 | P1 | 채용 또는 ML 엔지니어 육성 |
| **LPU 모니터링/관측 엔지니어** | LPU 메트릭 익스포터, TTFT·토큰/초 대시보드 | P1 | 내부 육성(기존 VictoriaMetrics 담당자) |

### 스킬 갭 요약

- **Groq SDK/런타임**: 현재 보유 없음 → POC 1–2주 + 공식 문서 기반 학습 필요
- **Slurm GRES 커스텀**: Slinky 사용 경험 있으나 GRES 타입 설계·디버깅 역량 부족
- **이종 가속기 스케줄링**: Kueue GPU 큐 경험만 있음 → GPU+LPU 이원 큐·라우팅 정책 신규 구축
- **LPU Device Plugin/CDI**: 공식 플러그인 미확인 → 커스텀 개발 역량 필요

## 3. 내부 교육 계획: 기존 GPU 엔지니어의 LPU 스택 재교육 로드맵

### Phase 1: 인지·개념 (1–2주)

| 항목 | 내용 | 대상 | 형태 |
|------|------|------|------|
| LPU 아키텍처 개요 | LPU vs GPU 차이, TTFT·대역폭 특성, 워크로드 분류 | 전체 플랫폼 엔지니어 | 2시간 워크샵 |
| NVIDIA–Groq 생태계 | 라이선스 구조, 2027 로드맵, ThakiCloud 전략적 위치 | 기술 리더·PM | 1시간 세션 |

### Phase 2: 기술 심화 (4–6주)

| 항목 | 내용 | 대상 | 형태 |
|------|------|------|------|
| Groq SDK/API 핸즈온 | Groq Cloud API 호출, 온프레미스 LPU 추론 파이프라인 | ML·인프라 엔지니어 2–3명 | 2일 집중 워크샵 + POC 참여 |
| Slinky GRES 심화 | `gres.conf` 커스텀, LPU 타입 등록, Slurm 플러그인 | Slinky 운영 담당자 | 1일 세션 + 문서 리딩 |
| Kueue·라우팅 정책 | extended resource, GPU/LPU 큐 설계, API 라우팅 로직 | 백엔드·스케줄링 담당자 | 1일 세션 + 스파이크 참여 |

### Phase 3: 운영·관측 (2–4주)

| 항목 | 내용 | 대상 | 형태 |
|------|------|------|------|
| LPU 메트릭·모니터링 | TTFT, 토큰/초, 대역폭 익스포터, VictoriaMetrics 연동 | SRE·모니터링 담당자 | 1일 세션 + 대시보드 구축 |
| 온콜·트러블슈팅 | LPU 노드 장애 대응, Device Plugin 이슈, 큐 대기 분석 | 온콜 로테이션 인력 | 런북 작성 + 1회 드릴 |

### 교육 일정 요약

- **총 예상 기간**: 8–12주 (Phase 1→2→3 순차)
- **집중 교육 일수**: 5–7일 (나머지는 자기주도 학습·POC 참여)
- **핵심 인원**: 5–8명 (LPU 파일럿 담당 그룹)

- **대안**: GPU/CUDA·Slurm·K8s 경험자 + Groq SDK 빠른 학습 가능 인재

- **리크루팅 채널**: Groq 커뮤니티, HPC/Slurm 밋업, NVIDIA 개발자 포럼, LinkedIn 타겟 광고

## 5. 조직 구조: GPU팀과 LPU팀 분리 vs 통합 운영 옵션

### 옵션 A: 분리 운영 (GPU팀 / LPU팀)

| 장점 | 단점 |
|------|------|
| 역할 명확, LPU 전문성 집중 | 인력 중복, 커뮤니케이션 비용 증가 |
| LPU 채용·육성 책임 소재 명확 | GPU/LPU 라우팅·통합 지점에서 협업 마찰 |
| LPU 실패 시 GPU팀 영향 최소화 | 총 인력 수 증가 필요 |

**적합**: LPU 규모가 크고 독립 사업 단위로 성장할 때

### 옵션 B: 통합 운영 (가속기 플랫폼팀)

| 장점 | 단점 |
|------|------|
| GPU/LPU 통합 라우팅·운영 일원화 | 개인당 스킬 범위 확대, 학습 부담 |
| 인력 효율, 온콜·트러블슈팅 공유 | LPU 전문성 심화 어려움 |
| 하이브리드 스케줄링 설계에 유리 | 핵심 인력 이탈 시 백업 부족 |

**적합**: 초기 파일럿·소규모 LPU 도입 단계

### 옵션 C: 하이브리드 (통합팀 + LPU 스페셜리스트 1–2명)

| 장점 | 단점 |
|------|------|
| 통합 운영 효율 + LPU 심화 역량 확보 | 스페셜리스트 의존도, 단일 장애점 |
| 단계적 확장 가능 (LPU 성장 시 팀 분리) | 스페셜리스트 채용·유지 비용 |

**권고**: 2026–2027 파일럿 단계에서는 **옵션 C** 권고. 가속기 플랫폼팀 내에 LPU 스페셜리스트 1–2명 배치, 나머지는 GPU 엔지니어가 LPU 기초 교육 후 통합 운영 참여.
