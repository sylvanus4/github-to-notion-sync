# Document Lifecycle Ground Rules

> Source: 문서 생애주기 그라운드 룰 셋업 미팅
> Status: Canonical reference for doc-lifecycle-auditor skill

## 공통 (Common)

1. **Notion vs GitHub 역할 분리**: 노션과 깃헙에 저장되는 문서의 역할을 나눈다.
2. **주간 문서 관리 시간**: 매주 금요일 오전 10:00 ~ 10:30 문서 관리 시간.
3. **오너쉽 지정 미팅**: 오너쉽이 비어있는 문서를 지정하기 위해 초기 수 회 미팅 진행.
4. **알림 봇 설정**: 노티를 위한 봇 설정은 @이상민 담당.
5. **승인자 불필요**: 문서 승인자 도입은 현재 불필요. 대신 모든 문서에 오너쉽 지정.
6. **삭제 우선**: 문서 관리의 첫 단계로 불필요 문서 삭제부터 진행.
7. **삭제 권한/기준**: 오너쉽 담당자 또는 레포 주인이 삭제. 기준은 각자 판단.

## 노션 (Notion)

1. **메타데이터**: 문서 생애주기 관리를 위한 기본 메타데이터 필수.
   양식까지 엄격하게 맞추지 않아도 됨.
   - 참고: [2-1 AI Platform IAM](https://www.notion.so/thakicloud/2-1-AI-Platform-IAM-2f79eddc34e680f1b2ecffa18ec692f8)
2. **단일 경로**: `ai-platform` 경로를 하나만 사용.
3. **아키텍처 노션 관리**: 아키텍처 문서는 노션으로 정리.
   - 독자(해림님, 의사결정권자 등)를 고려한 작성.

## 깃헙 / 레포 문서 (GitHub)

1. **docs 경로 컨벤션**:
   - 프론트엔드: 루트에 프로젝트 형태로.
   - 백엔드: 프로젝트 내 `docs/` 디렉토리로 관리.
2. **README 구조 문서화**: 디렉토리와 같은 구조적 정보는 README에 기재.
3. **SSoT 브랜치**: 단일진실원(SSoT)을 위한 브랜치 설정.
   - `dev`, `main` 등. 또는 `head` 사용.
   - 레포 주인이 설정 담당.

## 이후 액션 아이템 (Future Action Items)

1. **도메인별 아키텍처 다이어그램**: 도메인 담당자별 아키텍처를 그려서 노션에 공유.
2. **공통 포맷 제공**: 문서 관리가 자리잡으면 공통 포맷 제공 (PRD, One-pager, ADR 등).
3. **LLM 벤치마크 논의**: 추후 별도 논의 예정.
