# 주간 릴리즈 운영 플로우 가이드

> **화요일 취합 → 수요일 QA → 목요일 배포**
> 매주 반복되는 고정 사이클입니다.

---

## 역할 정의

| 역할 | 담당자 | 책임 범위 |
|---|---|---|
| **릴리즈 오너** | 효정님 | 전체 사이클 관리, 취합, 블로커 에스컬레이션, 최종 승인 |
| **앱 오너** | 각 앱 개발 담당자 | PR 준비, 라벨 부착, QA 대응, 롤백 계획 작성 |
| **QA 매니저** | QA 담당자 | 수요일 테스트 수행, Pass/Fail 판정, 재현 스텝 작성 |
| **커뮤니케이션 허브** | 혜원님 | 비즈니스팀 공유, 외부 이해관계자 공지, 일정 조율 |

---

## 월요일 — 준비 (Preparation)

### 누가: 앱 오너 (각 앱 개발자)

| 시간 | 작업 | 상세 |
|---|---|---|
| 종일 | PR 준비 및 라벨 부착 | 이번 주 목요일 배포 대상 PR에 아래 라벨을 모두 부착 |

### 앱 오너가 할 일

1. **배포 대상 PR에 라벨 부착**
   - `release:thu` — 이번 주 목요일 배포 대상
   - `app:ai-platform` 또는 `app:agent-studio` — 앱 구분
   - `risk:low` / `risk:medium` / `risk:high` — 위험도
   - `qa:needed` — QA 필요

2. **PR 본문에 5개 필수 섹션 작성** (템플릿 사용)
   - `## Changes` — 변경 사항
   - `## User Impact` — 사용자 영향
   - `## QA Method` — QA 방법
   - `## Rollback Method` — 롤백 방법
   - `## Related Issue/Ticket` — 관련 이슈

3. **PR에 본인을 Assignee로 지정**

> ⚠️ **화요일 오전까지 라벨이 없는 PR은 이번 주 배포에서 자동 제외됩니다.**

### 누가: 효정님 (릴리즈 오너)

| 시간 | 작업 |
|---|---|
| 오후 | `#release-control`에 리마인더 공지 — "화요일 오전까지 PR 라벨 부착 완료해주세요" |

## 화요일 — 취합 (Collection)

#### 누가: 효정님 (릴리즈 오너)

| 시간 | 작업 | 상세 |
|---|---|---|
| 10:00 AM | **취합 마감** | GitHub에서 `release:thu` 라벨이 달린 모든 PR 수집 |
| 10:00~11:00 | **검증** | 각 PR의 라벨, 템플릿 5섹션, Assignee 확인 |
| 11:00 | **Notion 업데이트** | `Weekly Release - {목요일 날짜}` 페이지 생성, 각 항목 등록 |
| 11:30 | **Slack 공지** | `#release-control`에 "주간 릴리즈 후보 목록 v1" 게시 |

#### 취합 결과 분류

| 상태 | 의미 | 다음 조치 |
|---|---|---|
| ✅ 완료 | 라벨 + 템플릿 + Assignee 모두 OK | 수요일 QA 대상 |
| ⚠️ 보완 필요 | 일부 누락 (라벨, 템플릿 섹션 등) | 앱 오너에게 보완 요청 → 수요일 QA 전까지 |
| ❌ 제외 | PR 없음 또는 치명적 누락 | 이번 주 배포에서 제외 |

### 오후: QA 준비

#### 누가: 앱 오너

| 시간 | 작업 |
|---|---|
| 오후 | 누락 항목 보완 (라벨, PR 본문 등) |
| 오후 | QA 환경 준비 (스테이징 배포, 테스트 데이터 등) |

#### 누가: 효정님

| 시간 | 작업 |
|---|---|
| 오후 | QA 대상 확정, 블로커 식별 |
| 오후 | 비즈니스팀 사전 공유 필요한 고영향 항목 식별 |

#### 누가: 혜원님 (커뮤니케이션 허브)

| 시간 | 작업 |
|---|---|
| 오후 | 고영향 항목 비즈니스팀 사전 공유 |

## 수요일 — QA (Quality Assurance)

### 종일: QA 수행

#### 누가: QA 매니저

| 시간 | 작업 | 상세 |
|---|---|---|
| 오전~오후 | **QA 수행** | 각 항목의 PR에 작성된 "QA Method"에 따라 테스트 |
| 수시 | **결과 기록** | Notion에 결과 기록: `Pass`, `Fail`, `Conditional Pass` |
| 수시 | **Fail 항목** | 재현 스텝 작성, Notion 상태를 `Hold`로 변경 |
| EOD | **GitHub 라벨 업데이트** | Pass 항목: `qa:needed` → `qa:done` 교체 |

#### QA 결과별 처리

| QA 결과 | Notion 상태 | GitHub 라벨 | 목요일 배포 |
|---|---|---|---|
| **Pass** | `QA Passed` → `Ready for Release` | `qa:done` | ✅ 배포 대상 |
| **Conditional Pass** | `QA Passed` (조건 기록) | `qa:done` | ✅ 조건부 배포 |
| **Fail** | `Hold` | `qa:needed` 유지 | ❌ 배포 제외 |
| **미완료** | `Hold` (자동) | `qa:needed` 유지 | ❌ 배포 제외 |

#### 누가: 앱 오너

| 시간 | 작업 |
|---|---|
| 수시 | QA 매니저 질의 응답, 이슈 대응 |
| 수시 | Fail 항목 수정 후 재QA 요청 |

#### 누가: 효정님

| 시간 | 작업 |
|---|---|
| EOD | **QA 게이트 적용** — 결과 없는 항목 자동 `Hold` 처리 |
| EOD | `#release-control`에 QA 현황 요약 게시 |
| EOD | 블로커 에스컬레이션 (필요 시 `#효정-의사결정`으로) |

> ⚠️ **수요일 종일까지 QA 결과가 없는 항목은 자동으로 Hold 처리되어 목요일 배포에서 제외됩니다.**

## 목요일 — 배포 (Deployment)

### 오전: 최종 확인 및 공지

#### 누가: 효정님 (릴리즈 오너)

| 시간 | 작업 | 상세 |
|---|---|---|
| 09:00 | **릴리즈 목록 잠금** | `Ready for Release` 항목만 최종 배포 목록에 포함 |
| 09:00 | **신규 항목 거절** | 목요일에 새로 추가하려는 항목은 거절 (핫픽스 제외) |
| 09:30 | **Notion 최종 업데이트** | 비즈니스팀/QA팀 공유 체크박스 확인 |
| 10:00 | **배포 예정 공지** | `#release-control`에 배포 시간, 항목 목록, 리스크 프로필 게시 |

#### 누가: 혜원님

| 시간 | 작업 |
|---|---|
| 오전 | 비즈니스팀 최종 공유, QA팀 배포 일정 안내 |

### 오후: 배포 실행

#### 누가: 앱 오너

| 시간 | 작업 |
|---|---|
| 14:00 (기본) | **배포 실행** — 본인 담당 앱 배포 |
| 배포 직후 | **프로덕션 확인** — 배포된 기능 정상 동작 확인 |
| 배포 직후 | **Notion 업데이트** — 상태를 `Released`로 변경 |

#### 누가: 효정님

| 시간 | 작업 |
|---|---|
| 배포 완료 후 | `#release-control`에 배포 완료 공지 |

| 18:00 | **회고 기록** — 이번 주 개선 포인트 3가지 기록, 쓰레드에 공유 |

### 배포 후 이슈 발생 시

| 상황 | 대응 |
|---|---|
| 경미한 이슈 | 다음 주 릴리즈에서 수정 |
| 심각한 이슈 | **핫픽스 프로세스 진입** → `#hotfix-alert` |

## 핫픽스 — 긴급 대응 (아무 요일)

### 누가: 앱 오너 (요청자)

| 순서 | 작업 | 상세 |
|---|---|---|
| 1 | **PR 생성** | `hotfix` 라벨 + `app:*` 라벨 부착 (⚠️ `release:thu`와 동시 사용 불가) |
| 2 | **PR 본문 작성** | 5섹션 템플릿 작성 + 고객/비즈니스 영향 필수 기재 |
| 3 | **Slack 알림** | `#hotfix-alert`에 핫픽스 요청 게시 |

### 누가: 효정님

| 순서 | 작업 |
|---|---|
| 4 | 핫픽스 PR 검증 (영향 기술서, 비즈니스 공유 여부 확인) |
| 5 | Notion Hotfix Queue에 등록 |

### 누가: QA 매니저

| 순서 | 작업 |
|---|---|
| 6 | **축약 QA** — 당일 내 완료 |
| 7 | 결과 기록 (Notion + `#hotfix-alert` 쓰레드) |

### 누가: 앱 오너

| 순서 | 작업 |
|---|---|
| 8 | QA 통과 후 **즉시 배포** |
| 9 | `#hotfix-alert`에 배포 완료/롤백 공지 |

> ⚠️ **핫픽스 3대 필수 요건**: ① 고객/비즈니스 영향 기술 ② 비즈니스팀 공유 여부 ③ QA 완료 여부

---

## 7대 핵심 룰 요약

| # | 룰 | 위반 시 |
|---|---|---|
| 1 | 화요일 오전까지 PR 없으면 → 이번 주 배포 제외 | 다음 주로 이월 |
| 2 | 수요일까지 QA 결과 없으면 → 목요일 배포 불가 | 자동 Hold |
| 3 | 핫픽스는 영향 기술서 + 공유 상태 필수 | 등록 거부 |
| 4 | 모든 배포 항목에 설명 가능한 앱 오너 필수 | 항목 제외 |
| 5 | 배포 당일 신규 항목 추가 금지 (핫픽스 제외) | 거절 |
| 6 | PR 템플릿 5섹션 필수 | 보완 요청 |
| 7 | `release:thu`와 `hotfix` 라벨 동시 사용 금지 | 라벨 수정 요청 |
