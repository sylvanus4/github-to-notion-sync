# QA 에이전트 설계 가이드

## 핵심 원칙: 경계면 교차 비교 (Integration Coherence Verification)

멀티 에이전트 시스템에서 가장 치명적인 결함은 **경계면 불일치 (Boundary Mismatch)**다.
각 에이전트의 출력이 개별적으로 유효해도 에이전트 간 데이터 연결이 깨지면 전체가 실패한다.

### 경계면 불일치 유형

| 유형 | 설명 | 예시 |
|---|---|---|
| **스키마 불일치** | 에이전트 A의 출력 형식 ≠ 에이전트 B의 기대 입력 | A가 JSON, B가 YAML 기대 |
| **의미 불일치** | 같은 필드명이 다른 의미로 사용 | `status`가 A에서는 HTTP 코드, B에서는 "active/inactive" |
| **누락 전달** | A가 생산한 데이터를 B가 무시하거나 참조 누락 | A의 분석 결과 중 일부가 B 보고서에 반영 안 됨 |
| **순서 가정** | A가 정렬된 데이터를 기대하는데 B가 비정렬 출력 | 우선순위 순 기대 vs 알파벳 순 출력 |

### QA 에이전트의 역할

QA 에이전트는 **개별 에이전트 출력의 존재 여부**보다 **에이전트 간 데이터 흐름의 일관성**을 검증한다.

```
[에이전트 A 출력] ←──비교──→ [에이전트 B 입력/출력]
        ↓                              ↓
    교차 검증: A의 출력이 B에서 올바르게 소비되었는가?
```

## QA 에이전트 설계

### subagent_type 선택

QA 에이전트는 **`generalPurpose`** 타입을 사용한다:
- 여러 파일을 읽고 비교해야 함 (Read 도구 필요)
- 검증 결과를 파일로 저장해야 함 (Write 도구 필요)
- 검증 스크립트를 실행해야 할 수 있음 (Shell 도구 필요)
- `explore` 타입은 쓰기 불가로 부적합

### model 선택

| 검증 복잡도 | model | 이유 |
|---|---|---|
| 형식 검증 (스키마, 필드 존재) | `fast` | 구조적 비교, 빠른 판단 |
| 의미 검증 (논리 일관성, 누락) | 기본 모델 | 맥락 이해 필요 |
| 복합 검증 (교차 참조, 추론) | 더 높은 모델 | 심층 분석 필요 |

## 검증 전략

### 1. 교차 비교 검증 (Cross-Comparison)

각 에이전트 쌍의 출력을 교차 비교:

```
에이전트 A 출력과 에이전트 B 출력을 동시에 읽고:
1. A가 생산한 항목이 B에서 참조되는가?
2. 참조된 항목의 값이 일치하는가?
3. A에만 있고 B에 없는 항목이 있는가? (누락)
4. B에만 있고 A에 없는 항목이 있는가? (환각)
```

### 2. 스키마 검증

각 Phase의 출력이 다음 Phase의 입력 요구사항을 충족하는지 검증:

```markdown
## Phase 1 → Phase 2 스키마 검증

| Phase 1 출력 필드 | Phase 2 기대 입력 | 일치 |
|---|---|---|
| `analysis.summary` (string) | `input.summary` (string) | ✅ |
| `analysis.risks` (array) | `input.risk_list` (array) | ⚠️ 필드명 불일치 |
| `analysis.score` (number) | — | ❌ Phase 2에서 미사용 |
```

### 3. 완결성 검증

최종 산출물이 모든 중간 산출물의 핵심 내용을 포함하는지 확인:

```
for each intermediate_output in [01_a.md, 01_b.md, 01_c.md]:
  for each key_finding in intermediate_output:
    assert key_finding is referenced in final_output
    if not: report as "누락 전달"
```

# {Domain} QA Agent

## 역할
개별 에이전트 출력이 아닌, 에이전트 간 데이터 흐름의 일관성을 검증한다.

## 입력
- `_workspace/01_*.md`: 모든 Phase 1 산출물
- `_workspace/02_*.md`: 모든 Phase 2 산출물 (있는 경우)
- `_workspace/final/*.md`: 최종 산출물

## 검증 항목

### 1. 교차 참조 완결성
- [ ] Phase 1의 모든 핵심 findings가 최종 산출물에 반영
- [ ] 에이전트 간 상충되는 결론이 명시적으로 처리됨

### 2. 스키마 일관성
- [ ] 각 Phase 출력 형식이 다음 Phase 입력과 호환
- [ ] 공통 필드의 값 범위/타입이 일치

### 3. 누락/환각 탐지
- [ ] 최종 산출물에 원본에 없는 내용 없음 (환각)
- [ ] 원본의 핵심 내용이 최종에 누락 없음

## 출력
- `_workspace/qa_report.md`: 검증 결과 보고서

## 증분 QA (Incremental QA)

전체 시스템을 한 번에 검증하지 말고, Phase별로 증분 검증한다:

```
Phase 1 완료 → QA(Phase 1 내부 일관성)
Phase 2 완료 → QA(Phase 1→2 경계면)
Phase 3 완료 → QA(Phase 2→3 경계면 + 전체 완결성)
```

장점:
- 조기 발견: Phase 1 오류를 Phase 3에서 발견하면 재작업 비용 증가
- 범위 축소: 각 검증이 소규모여서 정확도 향상
- 디버깅 용이: 어느 경계에서 불일치 발생했는지 즉시 파악

## 안티패턴

| 안티패턴 | 문제 | 올바른 접근 |
|---|---|---|
| 존재 여부만 확인 | 파일 있음 ≠ 내용 유효 | 내용 교차 비교 |
| 단일 에이전트 출력만 검증 | 개별 유효 ≠ 통합 유효 | 에이전트 쌍 교차 검증 |
| 마지막에만 QA | 재작업 비용 폭증 | Phase별 증분 QA |
| QA 에이전트에 explore 타입 | 검증 결과 저장 불가 | generalPurpose 사용 |
| 검증 기준 없이 "검수해줘" | 무엇을 검증할지 불명확 | 구체적 assertion 목록 제공 |
