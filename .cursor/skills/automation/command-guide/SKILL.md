# Command Guide (Interactive Discovery)

## Output Language

All outputs MUST be in Korean (한국어). Technical terms and command names remain in English.

## Purpose

Help the user pick the **right** slash command (`/command-name`) from 624 available commands across 42 categories. Three modes address different needs: browsing by domain, intent-based picking, and multi-command chain composition.

## Data Source

The authoritative command index lives at:

```
.cursor/skills/automation/command-guide/references/command-registry.json
```

This file is generated by `scripts/build-command-registry.mjs` and contains:
- `meta.totalCommands`: total command count
- `meta.categories`: category → count mapping
- `commands[]`: array of `{ name, description, argumentHint, category, subdir?, examples? }`

Always read this file at the start of every invocation. Do NOT rely on memorized command lists.

## Modes

### Mode 1: Browse (`--browse` or `--category`)

Explore commands by domain category.

**Activation triggers:**
- "커맨드 카테고리 보여줘", "browse commands", "list categories"
- "trading 관련 커맨드", "PM 커맨드 뭐 있어"
- User specifies a category name

**Procedure:**
1. Read `command-registry.json`
2. If no category specified: show all categories with counts, sorted by count descending
3. If category specified: list all commands in that category with name + one-line description
4. For each listed command, show `argumentHint` if available
5. Suggest related categories the user might also want to explore

**Output format:**
```
## 📂 {category} ({count}개 커맨드)

| 커맨드 | 설명 | 인자 힌트 |
|--------|------|-----------|
| /name  | ...  | ...       |

💡 관련 카테고리: {related1}, {related2}
```

### Mode 2: Pick (`--pick` or default)

Match user intent to the best command. This is the **default mode** when no flag is specified.

**Activation triggers:**
- "어떤 커맨드 써야 해?", "which command", "pick a command"
- "코드 리뷰하고 싶어", "PR 만들어야 해", "주식 분석하려면"
- Any task description without explicit mode flag

**Procedure:**
1. Read `command-registry.json`
2. **Capture intent** — Extract: task type, constraints, desired output
3. **Category shortlist** — Identify 2-3 most relevant categories from the user's intent
4. **Command match** — Within shortlisted categories, compute a **weighted score** per command, then apply boosts and deterministic tie-breakers:

   **A. Weighted base (additive)**
   Compute the integer sum `baseScore` from signal matches (each contributes at most once):
   - **Name pattern**: exact substring or high-confidence token overlap in `name` → **+3**
   - **Description**: keyword/phrase match in `description` → **+2**
   - **ArgumentHint**: relevance to stated inputs/constraints → **+1**

   **B. Category-boost multiplier**
   From step 3's **2–3** shortlisted categories, take the **top-2** confidence ordering as `{C1, C2}`. If `command.category` equals `C1` or `C2`, set `multipliedScore = baseScore * 1.5` (round the product to one decimal internally; when ranking, round to the nearest integer). Otherwise `multipliedScore = baseScore`.

   **C. Documentation (“recency”) bonus**
   Let `exampleCount` be `Array.isArray(examples) ? examples.length : 0`. If `exampleCount >= 2`, add **+1** to `multipliedScore` → `finalScore` (commands with multiple examples are treated as better documented / more reliable in the registry).

   **D. Tie-breaking** (when sorting candidates with equal `finalScore`, apply until unique):
   1. Prefer commands with **≥1** example over **0**; if both sides have examples, prefer **higher** `exampleCount`.
   2. Prefer **shorter** `name` string length (include the `/` prefix consistently for all candidates).
   3. Break any remaining tie by **ascending alphabetical** order on `name`.

5. **Compound intent check** — If the query contains 2+ verbs or tasks ("리뷰하고 커밋"), auto-switch to Chain mode instead of forcing a single pick.
6. **Confidence score** — Compute match confidence (High ≥5, Medium 3-4, Low ≤2). If Low: state "확신도 낮음 — 의도를 더 구체적으로 말씀해주세요" and ask a clarifying question instead of guessing.
7. **Rank and present** — Show:
   - **Primary recommendation**: 1 best command with full details and confidence level
   - **Alternatives**: 2 runner-up commands with trade-off explanation
   - **Anti-recommendations**: commands that sound similar but are wrong for this intent

**Output format:**
```
## 🎯 추천 커맨드

### 1순위: `/command-name`
- **설명**: {description}
- **사용법**: `/{command-name} {argumentHint}`
- **사용 예시**: {example or generated usage}
- **카테고리**: {category}

### 대안
- `/alt1` — {one-line trade-off explanation}
- `/alt2` — {one-line trade-off explanation}

### ⚠️ 이건 아닙니다
- `/wrong1` — {why this looks similar but is wrong}
```

### Mode 3: Chain (`--chain`)

Compose multi-command workflows for complex tasks.

**Activation triggers:**
- "커맨드 체인", "command chain", "여러 커맨드 조합"
- "코드 리뷰하고 커밋하고 PR까지", "분석하고 리포트까지"
- Task description implying multiple sequential steps

**Procedure:**
1. Read `command-registry.json`
2. Decompose the user's goal into 2-5 sequential steps
3. For each step, pick the best command using the Pick algorithm
4. Identify handoff artifacts between steps (file paths, branch names, report outputs)
5. Present the chain as a numbered recipe

**Output format:**
```
## 🔗 커맨드 체인: {goal summary}

| 단계 | 커맨드 | 목적 | 산출물 |
|------|--------|------|--------|
| 1    | /cmd1  | ...  | ...    |
| 2    | /cmd2  | ...  | ...    |
| 3    | /cmd3  | ...  | ...    |

### 실행 순서
1. `/{cmd1} {args}` — {what this does}
2. `/{cmd2} {args}` — {what this does, using output from step 1}
3. `/{cmd3} {args}` — {final step}

💡 팁: {practical advice about the chain}
```

## Context Signals

When the user's intent is ambiguous, use available workspace context to boost relevant categories before scoring:

| Signal | Detection Method | Category Boost |
|--------|------------------|----------------|
| Git branch name contains `fix/` or `hotfix/` | Check current branch via `git branch --show-current` | +2 to debugging, git-release |
| Recently edited `.test.` or `.spec.` files | Check recent editor tabs | +2 to testing |
| Terminal shows running dev server | Check terminal output | +1 to design-frontend, debugging |
| Current directory is under `backend/` | Check working directory | +1 to code-review, infrastructure |
| Current directory is under `thaki-ui/` | Check working directory | +1 to design-frontend |
| Files with `.md` extension recently opened | Check recent tabs | +1 to documents-office, pm-planning |
| Uncommitted changes exist | `git status --porcelain` | +1 to git-release |

Apply context boosts BEFORE the Pick scoring algorithm (Mode 2 step 3). Context boosts stack with category shortlisting but cap at +3 total boost per category.

If no context signals are detectable (e.g., fresh session), proceed without boosts — do NOT hallucinate context.

## Disambiguation Rules

When multiple commands share similar names or purposes:

1. **`/simplify` vs `/deep-review` vs `/code-review-all`**
   - `/simplify`: Quick single-pass review with auto-fix (best for small changes)
   - `/deep-review`: 4-agent parallel domain review (best for multi-file changes)
   - `/code-review-all`: Full adversarial 3-agent review (best for pre-release audit)

2. **`/domain-commit` vs `/ship` vs `/release-ship`**
   - `/domain-commit`: Split commits by domain only (no push, no PR)
   - `/release-ship`: Commit + push + PR without code review
   - `/ship`: Full pipeline: review + fix + commit + PR

3. **`/today` vs `/daily-stock-check` vs `/weekly-stock-update`**
   - `/today`: Full daily pipeline (sync + screen + analyze + report)
   - `/daily-stock-check`: Analysis only (no data sync)
   - `/weekly-stock-update`: Price data sync only (no analysis)

4. **`/morning` vs `/morning-ship` vs `/sod-ship`**
   - `/morning`: Google daily + stock pipeline
   - `/morning-ship`: Git pull all repos + Google + stock
   - `/sod-ship`: Git sync only (commit + push + pull all repos)

5. **`/kb-query` vs `/kb-search` vs `/cognee`**
   - `/kb-query`: Full Q&A synthesis with citation-backed answers from the wiki
   - `/kb-search`: Quick FTS5 or ripgrep lookup without Q&A synthesis
   - `/cognee`: Graph-based knowledge search across entities and relationships

## Procedure (All Modes)

1. **Read registry** — Load `command-registry.json` from references directory. If missing, report error and suggest running `node scripts/build-command-registry.mjs`.

2. **Detect mode** — Check for `--browse`, `--chain`, or default to `--pick`. Also detect mode from natural language ("카테고리 보여줘" → browse, "여러 개 조합" → chain).

3. **Execute mode** — Follow the mode-specific procedure above.

4. **Quality check** — Before responding:
   - Primary recommendation is a single clear command (not a vague list)
   - Alternatives explain trade-offs
   - Usage example is paste-ready
   - Anti-recommendations prevent common confusion
   - Category adjacency suggestions are included
   - Browse mode: cap output at 15 commands per category; say "외 N개 — `--browse {category}`로 전체 보기" for overflow
   - Chain mode: max 5 steps; if more needed, split into 2 chains

5. **Honest reporting** — Never inflate confidence. If the best match scores ≤2, say "정확히 일치하는 커맨드를 찾지 못했습니다" and show the closest 3 with explicit caveats. Never fabricate a command that doesn't exist in the registry.

6. **Ambiguity handling** — If intent is unclear after analyzing the query, ask up to **2** focused questions:
   - "입력은 무엇인가요?" (What is the input? — file, URL, branch, topic)
   - "결과물은 어디에 필요한가요?" (Where does the output go? — Slack, Notion, PR, local file)

7. **Verification** — After generating a recommendation, re-check that every recommended command name exists in `commands[]` of the loaded registry. Remove any that don't.

## Error Handling

- **Registry not found**: "⚠️ command-registry.json을 찾을 수 없습니다. `node scripts/build-command-registry.mjs`를 실행해주세요."
- **No matching commands**: Show top 3 closest commands by name similarity and suggest the user try `/skill-guide` for skill-based alternatives.
- **Ambiguous category**: List matching categories and ask the user to narrow down.

## Integration with skill-guide

- If the user's task is better served by a **skill** rather than a **command**, say so explicitly and redirect: "이 작업은 커맨드보다 스킬이 적합합니다. `skill-guide`를 사용해보세요."
- Commands are single-purpose invocations; skills are multi-step workflows. When the task requires orchestration, recommend skills.

## Gotchas

1. **Stale registry** — If `generatedAt` in `command-registry.json` is older than 7 days, warn: "레지스트리가 {N}일 전 생성되었습니다. `node scripts/build-command-registry.mjs`로 갱신을 권장합니다."
2. **Name collision** — Some commands exist in both root and `spec-kit` subdirectories (e.g., `/plan`). Always include `subdir` context when recommending spec-kit commands.
3. **Command ≠ Skill** — `/skill-guide` discovers skills; this skill discovers commands. If a user confuses the two, redirect without judgment.
4. **Compound intent misrouted** — "리뷰하고 커밋해줘" is a Chain, not a Pick. Detect multi-verb queries and auto-switch modes.
5. **Empty category** — If a category has 0 commands after filtering, say so explicitly rather than showing an empty table.
6. **Context signal staleness** — Workspace context (branch, files, terminal) reflects the current moment. Do not cache or carry over context from prior invocations. Re-detect on each run.

## Failure Scenarios

| Scenario | Response |
|----------|----------|
| Registry file missing or corrupted JSON | "⚠️ command-registry.json을 읽을 수 없습니다. `node scripts/build-command-registry.mjs`를 실행해주세요." — do NOT attempt to answer from memory. |
| User intent matches 0 commands (score=0 across all) | "해당 작업에 맞는 커맨드를 찾지 못했습니다. 스킬 기반 작업일 수 있습니다 — `skill-guide`를 사용해보세요." |
| User asks for a command that existed but was removed | "해당 커맨드는 현재 레지스트리에 없습니다. 삭제되었거나 이름이 변경되었을 수 있습니다." |
| Registry has 0 commands in a requested category | "'{category}' 카테고리에 등록된 커맨드가 없습니다. 관련 카테고리: {adjacents}" |

## Safety

- Do not expose file system paths, tokens, or internal IDs from the registry.
- Do not invent commands that are not in the registry. If a command doesn't exist, say so.
- If confidence is Low (≤2), never present the recommendation as definitive. Always qualify with uncertainty language.
