# 상태값 / 예외 / 제약사항 추출 가이드

`tech-doc-translator` 스킬의 **Identify / Enrich** 단계에서 사용하는 추출 규칙.
기술 문서에서 기획에 필요한 상태값, 예외 상황, 제약사항을 체계적으로 식별하고 정리하는 방법을 제공한다.

---

## 1. 상태값 추출

### 1.1 상태값 식별 기준

문서에서 다음 패턴이 나타나면 상태값으로 식별한다:

| 패턴 | 예시 | 추출 대상 |
|------|------|----------|
| enum / status 필드 | `status: "pending" \| "active" \| "deleted"` | 각 enum 값 |
| 상태 전이 다이어그램 | `DRAFT → REVIEW → APPROVED` | 각 노드 |
| HTTP 상태 코드 목록 | `200 OK`, `404 Not Found`, `429 Too Many Requests` | 사용자에게 의미 있는 코드 |
| 불리언 플래그 | `is_active`, `is_verified`, `has_permission` | true/false 각각의 의미 |
| 생명주기 설명 | "생성 후 ~ 승인되면 ~ 만료됩니다" | 각 단계 |
| 에러 코드 표 | `E001: Invalid Input`, `E002: Timeout` | 사용자에게 노출되는 에러 |

### 1.2 상태값 정리 형식

추출된 각 상태에 대해 4개 컬럼을 채운다:

```
| 상태명 | 의미 | 사용자에게 보이는 것 | 전환 조건 |
```

**컬럼별 작성 규칙**:

- **상태명**: 원문의 영문 상태명 + 한글 번역 (예: `PENDING (대기중)`)
- **의미**: 기술적 의미를 한 문장으로. 기술 용어 사용 최소화
- **사용자에게 보이는 것**: UI/UX 관점. "사용자가 화면에서 보는 것"
  - 구체적으로: 버튼 비활성화, 로딩 스피너, 안내 메시지 등
  - 상태가 UI에 노출되지 않는 경우: "사용자에게 직접 표시되지 않음 (내부 처리 상태)"
- **전환 조건**: "무엇이 발생하면 이 상태가 되는가"
  - 사용자 액션 기반: "사용자가 [행동]을 하면"
  - 시스템 기반: "서버가 [처리]를 완료하면"
  - 시간 기반: "[N]시간 경과 후 자동으로"

### 1.3 상태 전이 흐름 작성

상태들 간의 전이를 화살표로 시각화한다:

```
[초기] → (사용자 제출) → [대기중] → (관리자 승인) → [활성]
                                  → (관리자 반려) → [반려됨] → (사용자 수정) → [대기중]
                         [활성] → (유효기간 만료) → [만료됨]
                         [활성] → (사용자 삭제 요청) → [삭제 예정] → (30일 경과) → [삭제됨]
```

작성 규칙:
- 괄호 `()` 안에는 전환 조건 (트리거)
- 대괄호 `[]` 안에는 상태명
- 분기가 있으면 들여쓰기로 분리
- "되돌릴 수 있는" 전환과 "되돌릴 수 없는" 전환을 구분하여 표기

### 1.4 상태가 없는 문서

문서에 명시적 상태값이 없는 경우:
- 암묵적 상태를 추론하지 않음 (과도한 해석 방지)
- "해당 없음" 표기 + 이유 설명
- 단, "활성/비활성" 같은 자명한 on/off 상태는 추출

## 2. 예외 상황 추출

### 2.1 예외 식별 기준

| 문서 내 표현 | 예외 유형 | 기획 관점 |
|-------------|----------|----------|
| "~하면 에러가 발생합니다" | 입력 에러 | 입력 검증 UI 필요 |
| "~인 경우 실패합니다" | 처리 실패 | 실패 안내 메시지 필요 |
| "~은 지원하지 않습니다" | 기능 제한 | 해당 기능 비활성화 또는 안내 필요 |
| "~시 타임아웃됩니다" | 시간 초과 | 로딩 상태 + 타임아웃 안내 필요 |
| "~인 경우 재시도하세요" | 일시적 오류 | 자동 재시도 또는 재시도 버튼 |
| "~은 권한이 필요합니다" | 권한 부족 | 권한 안내 + 요청 절차 UI |
| "~의 최대값은 N입니다" | 한계 초과 | 입력 제한 UI + 초과 시 안내 |
| "동시에 ~할 수 없습니다" | 동시성 제한 | 동시 작업 방지 UI |

### 2.2 예외 정리 형식

```
| 예외 상황 | 발생 조건 | 기술적 결과 | 사용자에게 보이는 것 | 기획 대응 |
```

**기획 대응 컬럼 작성 규칙**:
- 구체적 UI/UX 대응 방안 제시 (예: "에러 토스트 메시지 표시")
- "사용자가 다음에 할 수 있는 행동" 명시
- 되돌릴 수 있는지 여부 표기

### 2.3 예외 우선순위 분류

추출된 예외를 사용자 영향도로 분류한다:

| 등급 | 기준 | 기획 시 대응 |
|------|------|-------------|
| **Critical** | 데이터 손실, 결제 오류, 서비스 중단 | 반드시 예방 로직 + 복구 절차 기획 |
| **High** | 주요 기능 사용 불가, 잘못된 결과 노출 | 명확한 에러 안내 + 대안 경로 기획 |
| **Medium** | 일부 기능 제한, 성능 저하 | 안내 메시지 + 예상 대기 시간 표시 |
| **Low** | UI 깨짐, 비핵심 기능 미작동 | 알려진 제한으로 문서화 |

## 3. 제약사항 추출

### 3.1 제약사항 식별 기준

| 문서 내 표현 | 제약 유형 | 추출 포인트 |
|-------------|----------|------------|
| 수치 제한 ("최대 N", "N까지만") | 용량/횟수 제한 | 정확한 숫자와 단위 |
| 지원 범위 ("~만 지원", "~는 미지원") | 호환성 제한 | 지원/미지원 목록 |
| 시간 제한 ("N초 내에", "N일 후 만료") | 시간 제약 | 정확한 시간과 초과 시 동작 |
| 순서 제약 ("A 후에만 B 가능") | 의존성 | 선행 조건 목록 |
| 동시성 제한 ("동시 N개까지") | 동시 접근 | 동시 사용 한도와 초과 시 동작 |
| 권한 제한 ("관리자만", "인증 필요") | 접근 제어 | 역할별 접근 매트릭스 |
| 비용 제한 ("N원/건", "무료 N건까지") | 비용 | 과금 기준과 임계값 |

### 3.2 제약사항 → 기획 영향 매핑

추출된 각 제약에 대해 3단계 매핑을 수행한다:

```
기술 제약 → 기획 시 고려사항 → 사용자 영향
```

**매핑 예시**:

```
기술 제약: "파일 업로드 최대 10MB"
→ 기획 시 고려사항: 업로드 전 파일 크기 검증 UI, 초과 시 안내 메시지, 압축 옵션 제공 여부
→ 사용자 영향: 고화질 이미지나 대용량 문서 업로드 시 실패 경험

기술 제약: "API 호출 분당 100회 제한"
→ 기획 시 고려사항: 대량 작업 시 진행률 표시, 제한 초과 시 대기 안내
→ 사용자 영향: 빠르게 반복 조작 시 일시적 제한, "잠시 후 다시 시도" 안내

기술 제약: "데이터 삭제 후 30일 보관, 이후 영구 삭제"
→ 기획 시 고려사항: 삭제 확인 모달, 복구 가능 기간 안내, 영구 삭제 경고
→ 사용자 영향: 30일 내 복구 가능, 이후 복구 불가
```

### 3.3 제약사항 체크리스트 자동 생성

추출된 제약사항을 기반으로 "기획 시 확인할 것" 체크리스트를 자동 생성한다:

```markdown
**기획 시 확인할 것**:
- [ ] 수치 제한에 대한 사용자 안내 문구 준비 여부
- [ ] 제한 초과 시 대안 경로 기획 여부
- [ ] 에러/실패 시 사용자 복구 방법 정의 여부
- [ ] 권한별 접근 범위 확인 및 비인가 접근 시 안내 여부
- [ ] 시간 제한에 대한 사용자 고지 방안 여부
```

## 4. 종합 분석 순서

기술 문서를 받으면 다음 순서로 분석한다:

1. **통독**: 전체 문서를 한 번 읽고 핵심 기능과 범위 파악
2. **상태값 스캔**: 1.1의 패턴으로 상태값 후보 식별
3. **예외 스캔**: 2.1의 패턴으로 예외 상황 식별
4. **제약 스캔**: 3.1의 패턴으로 제약사항 식별
5. **교차 검증**: 상태값 ↔ 예외 ↔ 제약 간 누락 확인
   - 각 상태 전이에서 예외가 발생할 수 있는가?
   - 각 제약이 특정 상태에서만 적용되는가?
6. **기획 영향 매핑**: 추출된 모든 항목에 기획 관점 영향 연결
7. **우선순위 분류**: 사용자 영향도 기준으로 Critical → Low 분류

### 교차 검증 체크포인트

| 검증 항목 | 질문 |
|-----------|------|
| 상태 → 예외 | "이 상태에서 실패하면 어떻게 되는가?" |
| 예외 → 상태 | "이 에러가 발생하면 상태가 어떻게 변하는가?" |
| 제약 → 상태 | "이 제약이 특정 상태에서만 적용되는가?" |
| 제약 → 예외 | "이 제약을 초과하면 어떤 에러가 발생하는가?" |
| 상태 + 예외 + 제약 | "기획서에서 빠뜨리기 쉬운 조합이 있는가?" |
