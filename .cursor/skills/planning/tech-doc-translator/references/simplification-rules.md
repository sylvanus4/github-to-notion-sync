# 기술 용어 → 비전문가 용어 변환 규칙

`tech-doc-translator` 스킬의 **Transform** 단계에서 사용하는 변환 규칙 모음.
기술 문서의 전문 용어를 비전문가가 이해할 수 있는 표현으로 바꾸는 체계적 규칙을 제공한다.

---

## 1. 변환 원칙

### P1: 비유 우선 (Analogy First)
기술 개념을 설명할 때 항상 일상적 비유를 먼저 제시한다.
비유는 독자가 이미 알고 있는 경험에 기반해야 한다.

```
나쁜 예: "API는 Application Programming Interface의 약자로, 소프트웨어 간 통신을 위한 인터페이스입니다."
좋은 예: "API는 '레스토랑 웨이터'와 같습니다. 손님(앱)이 주문(요청)을 하면, 웨이터(API)가 주방(서버)에 전달하고 음식(응답)을 가져다줍니다."
```

### P2: 시나리오 중심 (Scenario-Driven)
추상적 정의 대신 구체적 시나리오로 설명한다.

```
나쁜 예: "캐시는 자주 사용되는 데이터를 임시 저장하는 메커니즘입니다."
좋은 예: "사용자가 같은 페이지를 두 번째 열면, 처음보다 훨씬 빠르게 뜹니다. 이전에 받았던 내용을 '냉장고(캐시)'에 넣어뒀다가 꺼내 쓰기 때문입니다."
```

### P3: 계층적 설명 (Layered Explanation)
한 줄 요약 → 비유 → 상세 설명 순서로 제공한다.

```
1줄: "로드 밸런서는 트래픽을 여러 서버에 나눠주는 장치입니다."
비유: "마트 계산대가 여러 개 있을 때, 안내 직원이 줄이 짧은 곳으로 손님을 보내는 것과 같습니다."
상세: "웹사이트 방문자가 갑자기 늘어나면 한 서버가 모든 요청을 처리할 수 없습니다. 로드 밸런서는..."
```

### P4: 기획 영향 연결 (Planning Impact Link)
기술 개념 설명 후 반드시 "기획 시 의미하는 바"를 추가한다.

```
기술: "Rate Limit이 분당 100회로 설정되어 있습니다."
기획 영향: "→ 기획 시: 사용자가 1분 안에 100번 이상 새로고침하면 '잠시 후 다시 시도해주세요' 안내가 필요합니다."
```

## 2. 공통 기술 용어 변환 테이블

### 인프라/클라우드

| 기술 용어 | 비전문가 표현 | 비유 |
|-----------|-------------|------|
| Server | 서비스를 실행하는 컴퓨터 | 24시간 운영하는 가게 |
| Load Balancer | 접속자를 여러 서버에 나눠주는 장치 | 마트 계산대 안내 직원 |
| CDN | 전 세계에 콘텐츠 사본을 배치해 빠르게 전달하는 네트워크 | 전국 편의점 체인 — 가까운 곳에서 바로 받기 |
| Auto Scaling | 접속자 수에 따라 서버를 자동으로 늘리거나 줄이는 것 | 손님 많으면 계산대 추가 오픈, 한가하면 줄이기 |
| Container | 앱을 실행에 필요한 모든 것과 함께 묶은 패키지 | 이사할 때 짐을 컨테이너 박스에 넣는 것 |
| Kubernetes (K8s) | 컨테이너들을 자동으로 관리하는 시스템 | 대형 물류센터의 자동화 관리 시스템 |
| Region | 서비스가 실행되는 물리적 지역 | 지사/지점의 위치 |
| Availability Zone | 같은 지역 내 독립된 데이터센터 | 같은 도시 내 다른 건물의 백업 사무실 |
| Failover | 장애 시 자동으로 예비 시스템으로 전환 | 정전 시 비상 발전기 자동 가동 |
| Latency | 요청 후 응답까지 걸리는 시간 | 버튼 누른 후 반응까지의 대기 시간 |
| Throughput | 단위 시간당 처리 가능한 작업량 | 고속도로의 차선 수 (넓을수록 많이 통과) |
| SLA | 서비스 품질 보장 약속 | "99.9% 시간 동안 정상 운영" 보증서 |

### API/통신

| 기술 용어 | 비전문가 표현 | 비유 |
|-----------|-------------|------|
| API | 프로그램 간 데이터를 주고받는 창구 | 레스토랑 웨이터 |
| Endpoint | API의 특정 기능을 호출하는 주소 | 레스토랑 메뉴의 각 항목 |
| Request/Response | 요청과 응답 | 주문과 음식 서빙 |
| REST API | 웹 표준을 따르는 API 방식 | 표준 주문서 양식 |
| Authentication | 사용자 신원 확인 | 건물 출입증 확인 |
| Authorization | 사용자 권한 확인 | 출입증으로 특정 층만 접근 가능 |
| Token | 인증 후 발급받는 임시 통행증 | 놀이공원 자유이용권 팔찌 |
| Rate Limit | 일정 시간 내 허용되는 최대 요청 수 | 1분에 엘리베이터 버튼 최대 100번까지만 인식 |
| Webhook | 이벤트 발생 시 자동으로 알림을 보내는 장치 | 택배 도착 알림 문자 |
| Timeout | 응답을 기다리는 최대 시간 | 전화 신호 30초 후 자동 끊김 |
| HTTP Status Code | 요청 결과를 숫자로 알려주는 코드 | 200=성공, 404=못찾음, 500=서버오류 |
| Pagination | 데이터를 페이지 단위로 나눠 전달 | 검색 결과의 1페이지, 2페이지... |

### 데이터/저장

| 기술 용어 | 비전문가 표현 | 비유 |
|-----------|-------------|------|
| Database | 데이터를 구조적으로 저장하는 공간 | 체계적으로 정리된 서류 캐비닛 |
| Cache | 자주 쓰는 데이터를 빠르게 꺼내쓰는 임시 저장소 | 책상 위에 올려둔 자주 보는 서류 |
| Queue | 작업을 순서대로 대기시키는 구조 | 은행 번호표 시스템 |
| Migration | 데이터나 시스템을 한 곳에서 다른 곳으로 옮기는 작업 | 사무실 이사 |
| Backup | 데이터 복사본을 별도로 보관 | 중요 서류의 사본을 금고에 보관 |
| Replication | 같은 데이터를 여러 곳에 동시에 유지 | 같은 서류를 서울 본사와 부산 지사에 동시에 보관 |
| Schema | 데이터의 구조/형식 정의 | 입사 지원서 양식 (이름, 나이, 연락처 칸이 정해져 있는 것) |
| Index | 데이터를 빠르게 찾기 위한 목차 | 책의 색인(찾아보기) |

### 보안/정책

| 기술 용어 | 비전문가 표현 | 비유 |
|-----------|-------------|------|
| Encryption | 데이터를 읽을 수 없는 형태로 변환 | 편지를 암호로 쓰기 |
| SSL/TLS | 통신 내용을 암호화하는 보안 프로토콜 | 봉인된 등기우편 |
| Firewall | 허용되지 않은 접근을 차단하는 보안 장치 | 건물 출입 게이트 |
| CORS | 다른 도메인에서의 접근을 제어하는 정책 | 본사 직원만 지사 자료에 접근 가능 |
| RBAC | 역할에 따라 접근 권한을 부여하는 방식 | 부서별로 다른 층 출입 권한 |
| Audit Log | 시스템 사용 기록을 남기는 로그 | CCTV 녹화 기록 |

### 상태/프로세스

| 기술 용어 | 비전문가 표현 | 비유 |
|-----------|-------------|------|
| State | 현재 상태/단계 | 주문의 현재 진행 상황 |
| State Machine | 정해진 규칙에 따라 상태가 변하는 구조 | 자판기: 동전 투입 → 음료 선택 → 배출 |
| Callback | 작업 완료 후 자동으로 실행되는 후속 동작 | "요리 다 되면 벨 울리기" |
| Polling | 주기적으로 상태를 확인하는 방식 | 5분마다 택배 추적 새로고침 |
| Event-Driven | 이벤트 발생 시에만 반응하는 방식 | 초인종이 울려야 문 여는 것 |
| Async | 결과를 기다리지 않고 다음 작업을 진행하는 방식 | 카페에서 주문 후 진동벨 받고 자리에서 대기 |
| Sync | 결과가 올 때까지 기다리는 방식 | 약국에서 조제 끝날 때까지 카운터 앞에서 대기 |
| Idempotent | 같은 요청을 여러 번 해도 결과가 동일 | 엘리베이터 버튼을 여러 번 눌러도 한 번 오는 것 |

## 3. 비유 생성 패턴

기존 테이블에 없는 용어를 만났을 때, 다음 패턴으로 비유를 생성한다.

### 패턴 A: 장소 비유
기술 인프라 → 물리적 건물/장소로 대응

```
서버 → 가게, 데이터센터 → 대형 창고, 네트워크 → 도로
```

### 패턴 B: 일상 행위 비유
기술 프로세스 → 일상적 행동으로 대응

```
인증 → 신분증 확인, 배포 → 이사, 모니터링 → CCTV 확인
```

### 패턴 C: 우편/택배 비유
데이터 전송 → 물건 배달로 대응

```
패킷 → 택배 박스, 라우팅 → 물류 경로 선택, 프로토콜 → 배송 규칙
```

### 패턴 D: 사무실 비유
시스템 구조 → 회사 조직으로 대응

```
마이크로서비스 → 부서별 업무 분담, API Gateway → 안내 데스크, 로깅 → 업무 일지
```

### 패턴 E: 음식점 비유
요청-응답 패턴 → 주문-서빙으로 대응

```
API 호출 → 주문, 응답 → 서빙, 큐 → 주문 대기열, 타임아웃 → 30분 후 주문 자동 취소
```

## 4. 변환 시 금지사항

| 금지 | 이유 | 대안 |
|------|------|------|
| 약어만 풀어쓰기 | "API = Application Programming Interface"는 설명이 아님 | 비유 + 시나리오로 설명 |
| 기술 용어로 기술 용어 설명 | "프록시는 중간 서버입니다"는 순환 설명 | "대리인"처럼 일상어로 설명 |
| "단순히 말하면" 후 여전히 어려운 설명 | 독자 혼란 | 실제로 쉬운 표현 사용 |
| 비유 없이 정의만 나열 | 기억에 남지 않음 | 반드시 비유 1개 이상 포함 |
| 한 용어에 비유 3개 이상 | 오히려 혼란 | 가장 직관적인 비유 1-2개 선택 |
| 과도한 주석/괄호 | 가독성 저하 | 비유는 인용 블록(`>`)으로 분리 |

## 5. 문서 유형별 변환 초점

| 문서 유형 | 변환 시 가장 중요한 것 | 비유 카테고리 |
|-----------|---------------------|-------------|
| API 문서 | 요청-응답 흐름, 에러 코드 | 음식점 비유 (주문-서빙) |
| 인프라 설계 | 구성 요소 간 관계, 장애 시나리오 | 장소/건물 비유 |
| 요구사항 | 상태 전이, 비즈니스 룰 | 일상 행위 비유 |
| 정책 문서 | 제한/금지 사항, 위반 결과 | 사무실 규칙 비유 |
| 기술 사양 | 수치 한계, 호환성 | 우편/택배 비유 |
