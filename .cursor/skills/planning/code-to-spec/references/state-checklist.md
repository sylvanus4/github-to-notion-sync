# 상태값 점검 체크리스트

기획서 생성 후 자가 점검에 사용하는 체크리스트. 모든 화면은 아래 필수 상태를 정의해야 한다.

## 필수 상태 (모든 데이터 표시 화면)

| 상태 | 설명 | 체크 포인트 |
|------|------|-----------|
| loading | 데이터 로딩 중 | 스피너/스켈레톤/프로그레스 바 중 하나 명시 |
| error | 에러 발생 | 에러 메시지 + 재시도 방법 명시 |
| empty | 데이터 없음 | 빈 상태 메시지 + 행동 유도(CTA) 명시 |
| success | 정상 표시 | 데이터가 있을 때의 기본 화면 명시 |

## 권장 상태 (상황별 추가)

| 상태 | 적용 조건 | 체크 포인트 |
|------|----------|-----------|
| unauthorized | 권한 체크가 있는 화면 | 권한 없음 메시지 + 대안 안내 |
| offline | 오프라인 대응이 필요한 앱 | 오프라인 상태 안내 + 캐시 데이터 표시 여부 |
| timeout | 장시간 요청이 있는 화면 | 타임아웃 메시지 + 재시도 안내 |
| rate_limited | API 호출 제한이 있는 기능 | 제한 초과 메시지 + 대기 시간 안내 |
| maintenance | 서비스 점검 가능성 | 점검 안내 메시지 |
| partial_error | 여러 API를 조합하는 화면 | 부분 에러 시 표시 전략 (성공 데이터만 표시 등) |

## 인터랙션 상태

| 상태 | 적용 조건 | 체크 포인트 |
|------|----------|-----------|
| submitting | 폼 제출이 있는 화면 | 제출 중 버튼 비활성화 + 로딩 인디케이터 |
| validating | 입력값 검증이 있는 화면 | 검증 에러 메시지 위치 + 내용 |
| confirming | 삭제/중요 액션이 있는 화면 | 확인 모달/다이얼로그 내용 |
| success_feedback | 성공 피드백이 필요한 액션 | 토스트/알림 메시지 내용 |

## 품질 기준

| 기준 | 조건 | 판정 |
|------|------|------|
| 필수 상태 커버리지 | 모든 화면에 loading/error/empty/success 정의 | PASS: 100%, FAIL: < 100% |
| 예외 상황 문서화 | 예외 상황 3개 이상 | PASS: >= 3, FAIL: < 3 |
| 에러 경로 포함 | 사용자 플로우에 에러 시나리오 포함 | PASS: 있음, FAIL: 없음 |
| 기획 용어 사용 | 기술 용어(useState, useEffect 등) 미포함 | PASS: 없음, FAIL: 있음 |
| 한국어 작성 | 본문 전체 한국어 (기술 용어 제외) | PASS: 100%, FAIL: < 100% |
