# Risk Categories

기획안 기술 평가 시 사용하는 리스크 분류 체계.

## 리스크 유형

### 1. 기술 리스크

| 리스크 | 설명 | 탐지 기준 |
|--------|------|----------|
| 기술 성숙도 | 검증되지 않은 기술 사용 | GA 미달, 베타/프리뷰 서비스 |
| 벤더 종속 | 특정 클라우드에 과도한 의존 | 멀티 클라우드 전환 불가 서비스 |
| 통합 복잡도 | 다수 시스템 연동의 복잡성 | 5개 이상 외부 연동 |
| 성능 병목 | 성능 요구사항 대비 기술 한계 | 요구 latency < 서비스 기본 latency |

### 2. 운영 리스크

| 리스크 | 설명 | 탐지 기준 |
|--------|------|----------|
| 운영 복잡도 | 운영에 전문 인력 필요 | 자체 관리 인프라 사용 |
| 장애 복구 | 장애 시 복구 시간이 긴 경우 | 단일 장애 지점(SPOF) 존재 |
| 모니터링 부재 | 문제 감지가 어려운 구조 | 로깅/알림 미설계 |
| 스케일링 한계 | 트래픽 급증 대응 불가 | 수동 스케일링만 가능 |

### 3. 비용 리스크

| 리스크 | 설명 | 탐지 기준 |
|--------|------|----------|
| 비용 폭증 | 사용량 증가 시 비선형 비용 증가 | 데이터 전송 비용, API 호출 비용 |
| 예측 불가 | 월 비용을 예측할 수 없는 구조 | 사용량 기반 과금 + 변동폭 큰 트래픽 |
| 숨은 비용 | 명시적이지 않은 추가 비용 | 데이터 전송, 스토리지 IO, 로그 저장 |

### 4. 보안 리스크

| 리스크 | 설명 | 탐지 기준 |
|--------|------|----------|
| 데이터 유출 | 민감 데이터 노출 가능성 | 개인정보 처리 + 암호화 미설계 |
| 인증 취약 | 인증/인가 체계 미비 | 단순 API Key만 사용 |
| 규정 위반 | 데이터 규정 미준수 | 리전 요구사항 미충족 |

### 5. 일정 리스크

| 리스크 | 설명 | 탐지 기준 |
|--------|------|----------|
| 추정 초과 | 예상보다 오래 걸리는 개발 | 유사 사례 대비 짧은 일정 |
| 기술 학습 | 팀이 새 기술을 학습해야 함 | 팀 경험 없는 기술 스택 |
| 의존성 지연 | 외부 의존성으로 인한 블로킹 | 외부 API, 서드파티 승인 필요 |

## 리스크 완화 전략 템플릿

각 리스크에 대해 다음 구조로 완화 전략을 제시:

```
리스크: [리스크명]
확률: 높음/중간/낮음
영향: 높음/중간/낮음
등급: Critical/High/Medium/Low

완화 전략:
1. [예방 조치] — 리스크가 발생하지 않도록
2. [감지 방법] — 리스크 발생을 빠르게 인지
3. [대응 계획] — 발생 시 대응 방법
4. [대안] — 리스크가 감당 불가 시 대안 경로
```

## 기술팀 확인 필요 기준

AI 분석으로 판단할 수 없어 기술팀 확인이 반드시 필요한 항목:

- 내부 네트워크/보안 정책 관련 제약
- 기존 시스템의 실제 성능 수치
- 팀의 기술 스택 숙련도
- 내부 DevOps/인프라 운영 역량
- 기존 데이터 파이프라인의 정확한 구조
- 내부 규정/컴플라이언스 요구사항
- 라이선스 및 계약 제약
