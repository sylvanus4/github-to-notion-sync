# 기획서 품질 검증 체크리스트

## 1. 필수 섹션 존재 여부

각 기획서에 반드시 포함되어야 하는 섹션 목록. 제목이 정확히 일치하지 않아도 동일 의미의 섹션이면 통과.

| 섹션 | 동의어 | 심각도 | 판정 기준 |
|------|--------|--------|----------|
| 화면 개요 | 개요, Overview, 소개 | Critical | 화면의 목적과 접근 경로가 명시 |
| 기능 목록 | Features, 기능 정의, 기능 명세 | Critical | 테이블 형태로 기능이 나열 |
| 상태별 화면 | States, 화면 상태, UI States | Critical | loading/error/empty/success 포함 |
| 예외 상황 | Edge Cases, 예외 처리, 예외 케이스 | High | 최소 3개 예외 항목 |
| 사용자 플로우 | User Flow, 시나리오, 사용 시나리오 | High | 최소 1개 happy path |
| API 연동 | API, 서버 연동, 데이터 연동 | Medium | API를 사용하는 기능이 있을 때 |
| 변경 이력 | Changelog, 이력, History | Medium | 날짜+변경내용 테이블 |

## 2. 상태 커버리지 검증

### 검색 패턴

기획서 내에서 아래 키워드를 포함하는 섹션/테이블 탐색:

- "상태별", "States", "로딩", "에러", "빈 상태", "정상"
- 테이블 내 "loading", "error", "empty", "success" 셀

### 판정 매트릭스

| 정의된 상태 수 | 심각도 | 판정 |
|-------------|--------|------|
| 4/4 (loading + error + empty + success) | - | PASS |
| 3/4 | Medium | 누락 상태 명시 |
| 2/4 | High | 심각한 누락 |
| 1/4 이하 | Critical | 상태 정의 거의 없음 |

### 화면 유형별 필수 상태

| 화면 유형 | 필수 상태 | 추가 권장 상태 |
|----------|----------|-------------|
| 목록 화면 | loading, error, empty, success | 검색 결과 없음, 필터 결과 없음 |
| 상세 화면 | loading, error, success | 데이터 삭제됨, 권한 없음 |
| 폼 화면 | success, validating, submitting | 제출 에러, 네트워크 에러 |
| 대시보드 | loading, error, success | 부분 에러 (일부 위젯만 실패) |

## 3. 예외 상황 검증

### 필수 예외 카테고리

최소 아래 카테고리 중 3개 이상이 문서화되어야 함:

| 카테고리 | 예시 |
|---------|------|
| 네트워크 에러 | 서버 응답 없음, 타임아웃 |
| 권한 에러 | 로그인 만료, 접근 권한 없음 |
| 데이터 유효성 | 입력값 오류, 중복 데이터 |
| 비즈니스 규칙 | 재고 부족, 한도 초과 |
| 동시성 | 다른 사용자가 먼저 수정 |
| 시스템 | 서비스 점검, 기능 비활성화 |

### 예외 항목 완전성

각 예외 항목은 아래 3가지를 모두 포함해야 "완전":
1. **발생 조건** — 언제 이 예외가 발생하는가
2. **처리 방식** — 시스템이 어떻게 대응하는가
3. **사용자 메시지** — 사용자에게 뭐라고 안내하는가

## 4. 정책 정합성 검증

정책 문서가 제공된 경우, 아래 항목 대조:

| 대조 항목 | 방법 |
|----------|------|
| 문구 일치 | 정책에 정의된 고정 문구가 기획서에 동일하게 사용되는지 |
| 동작 일치 | 정책에 정의된 동작 규칙이 기획서에 반영되는지 |
| 제한 조건 | 정책의 제한 사항이 기획서에 명시되는지 |
| 충돌 감지 | 기획서 내용이 정책과 모순되는 부분이 있는지 |

## 5. 용어 일관성 검증

[terminology-guide.md](terminology-guide.md) 기반으로:

| 검증 유형 | 심각도 | 예시 |
|----------|--------|------|
| 동의어 혼용 | Medium | "주문"과 "오더"를 같은 문서에서 혼용 |
| 기술 용어 사용 | High | "useState", "useEffect" 등 코드 용어가 기획서에 등장 |
| 비표준 용어 | Low | 팀 합의 용어와 다른 표현 사용 |

## 6. API 정합성 검증

API 명세(Swagger/OpenAPI)가 제공된 경우:

| 검증 유형 | 심각도 |
|----------|--------|
| 기획서에 API가 명시되었지만 명세에 없음 | High |
| 명세에 API가 있지만 기획서에 없음 | Medium |
| 엔드포인트 경로 불일치 | High |
| HTTP 메서드 불일치 | High |
| 파라미터 불일치 | Medium |
