# Collaboration Structure Guide

협업 구조 점검 가이드. doc-quality-gate (선택 심화 점검)에서 사용.

## 역할별 전달력 체크리스트

### 개발팀 전달력

문서를 읽고 바로 구현에 착수할 수 있는가?

| # | 항목 | Weight | 설명 |
|---|------|--------|------|
| D1 | 상태 다이어그램 | 3 | 객체의 상태 전이가 시각적으로 명시됨 |
| D2 | API 계약 | 3 | 엔드포인트, 요청/응답 형식, 에러 코드 |
| D3 | 에러 처리 명세 | 3 | 실패 시나리오별 대응 방법 |
| D4 | 데이터 모델 | 2 | 필드명, 타입, 제약 조건 |
| D5 | 비즈니스 로직 규칙 | 2 | 계산식, 조건문, 우선순위 |
| D6 | 성능 요구사항 | 1 | 응답 시간, 동시 접속, 처리량 |
| D7 | 보안 요구사항 | 1 | 인증, 인가, 암호화 |

**점수**: Σ(존재 항목 × Weight) / Σ(전체 항목 × Weight) × 100%

## 구조 재구성 원칙

### 원칙 1: 역할 태그 분리

문서 섹션에 대상 역할 태그를 부여합니다:
- `[공통]` — 모든 역할이 읽어야 하는 내용
- `[개발]` — 개발팀 주요 참조 내용
- `[디자인]` — 디자인팀 주요 참조 내용
- `[QA]` — QA팀 주요 참조 내용
- `[기획]` — 기획팀 내부 참조 내용

### 원칙 2: 정보 계층 구조

```
1. TL;DR (전체 요약, 30초 읽기)
2. 개요 (배경, 목적, 범위)
3. 핵심 결정사항 (의사결정 로그)
4. 상세 명세 (역할별 태그 포함)
   4.1 기능 흐름 [공통]
   4.2 화면/UI 정의 [디자인]
   4.3 데이터/API 정의 [개발]
   4.4 예외 처리 [개발][QA]
   4.5 테스트 기준 [QA]
5. 부록 (용어집, 참고 자료, 변경 이력)
```

### 원칙 3: 상호참조 일관성

- 같은 객체/개념을 참조하는 섹션 간 용어 일치 검증
- 상태값 테이블과 화면 흐름 간 상태 매핑 일치 검증
- API 명세와 데이터 모델 간 필드명 일치 검증

## 종합 점수 계산

```
협업 구조 점수 = (개발 전달력 + 디자인 전달력 + QA 전달력) / 3 × 25 / 100
```

문서 유형별 가중치 조정:
- `spec`/`prd`: 세 역할 균등 (1:1:1)
- `policy`: 개발 2 : 디자인 0.5 : QA 1.5
- `design-guide`: 개발 1 : 디자인 2 : QA 1
